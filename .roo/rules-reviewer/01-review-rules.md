# Reviewer 모드 규칙

## 리뷰 원칙

1. **건설적 피드백**: 문제점만 지적하지 않고, 개선 방안을 함께 제시한다.
2. **근거 제시**: "이것은 문제다"가 아니라 "이것이 문제인 이유는..."으로 설명한다.
3. **심각도 분류**: 모든 이슈에 심각도를 표시하여 우선순위를 명확히 한다.

## 심각도 분류

| 심각도 | 설명 | 예시 |
|--------|------|------|
| **Critical** | 반드시 수정. 보안 취약점, 데이터 손실 가능성 | SQL 인젝션, 인증 우회 |
| **Major** | 수정 권장. 버그, 성능 문제, 잘못된 로직 | 무한 루프, 메모리 누수, 경쟁 조건 |
| **Minor** | 개선 권장. 가독성, 컨벤션 위반 | 네이밍 불일치, 중복 코드 |
| **Suggestion** | 선택사항. 더 나은 방법 제안 | 더 적합한 자료구조, 리팩토링 기회 |

## 리뷰 체크리스트

### 정확성
- 비즈니스 로직이 올바른가?
- 엣지 케이스가 처리되었는가?
- 에러 처리가 적절한가?

### 보안
- 사용자 입력이 검증/살균되는가?
- 인증/인가 검사가 올바른가?
- 민감 정보가 노출되지 않는가?

### 성능
- 불필요한 반복/계산이 없는가?
- N+1 쿼리 같은 패턴이 없는가?
- 대량 데이터에서도 동작하는가?

### 가독성
- 함수/변수 이름이 의도를 명확히 전달하는가?
- 복잡한 로직에 주석이 있는가?
- 프로젝트 컨벤션을 따르는가?

### 테스트
- 변경된 로직에 대한 테스트가 있는가?
- 테스트가 의미 있는 시나리오를 검증하는가?

## 출력 형식

리뷰 결과는 다음 형식으로 보고한다:

```
## 리뷰 요약

총 N개 이슈 발견 (Critical: N, Major: N, Minor: N, Suggestion: N)

### Critical
- [파일:줄] 이슈 설명 → 개선 방안

### Major
- [파일:줄] 이슈 설명 → 개선 방안

...
```
