## 아키텍처 결정 기준

1. **모듈 분리 원칙**: 각 모듈은 단일 책임을 가진다. 새로운 기능이 기존 모듈 범위를 벗어나면 별도 모듈 생성을 검토한다.
2. **의존성 최소화**: 모듈 간 의존성은 최소화한다. 인터페이스를 통한 느슨한 결합을 지향한다.
3. **인터페이스 우선**: 구현보다 인터페이스를 먼저 설계한다. 구현은 교체 가능하도록 설계한다.

## 계층 구조

<!-- 프로젝트의 실제 계층 구조로 교체 -->

```
Presentation Layer    (routes, controllers, views)
       ↓
Business Logic Layer  (services, use cases)
       ↓
Data Access Layer     (repositories, models)
       ↓
Infrastructure        (database, external APIs, file system)
```

- 상위 계층은 하위 계층에 의존한다 (역방향 금지).
- 같은 계층 간 횡단 의존은 최소화한다.

## 설계 원칙

- **SOLID 원칙** 준수
- **DRY**: 중복 코드는 공통 유틸리티로 추출
- **YAGNI**: 현재 필요하지 않은 기능은 미리 만들지 않는다
- **관심사 분리**: 비즈니스 로직, 데이터 접근, 프레젠테이션을 명확히 분리

## 새 모듈 추가 시 체크리스트

1. 기존 모듈로 해결할 수 없는가?
2. 모듈의 책임 범위가 명확한가?
3. 의존성 방향이 계층 규칙을 따르는가?
4. 인터페이스가 정의되었는가?
5. 테스트 전략이 수립되었는가?
